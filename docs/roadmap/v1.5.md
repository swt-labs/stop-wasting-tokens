# v1.5 roadmap

This is the canonical v1.5 roadmap. The docs-site landing page at [`v1-5-roadmap/index.mdx`](../v1-5-roadmap/index) is the user-facing summary; this document is the engineering plan with per-feature scope, dependencies, complexity estimates, and success criteria.

Format: each feature is `Fn` (where `n` is a stable identifier — references in PLAN.md / SUMMARY.md should cite `F2` rather than "the Claude Code driver" so renames don't break cross-references).

---

## Runtime

### F1 — Real Codex AgentSpawner wiring

**Scope.** Replace `@swt-labs/codex-driver`'s mock AgentSpawner with a live driver wrapping the Codex CLI's subagent-spawn API. Live `swt vibe --execute` runs unblocked. Per-role model resolution honored via the existing `resolve-agent-settings.sh`-equivalent path.

**Dependencies.** Codex CLI exposes a stable subagent-spawn API. (At v1.0 cut, `codex exec` is the closest equivalent — F1 may need to wrap that or wait for an official subagent surface.)

**Complexity.** L (~3-4 weeks)

**Success criteria.**
- Live `swt vibe --execute` against a real project completes without throws
- Mock driver retained for vitest hermetic runs (existing `test/mock-driver.ts` ScriptedPrompter pattern preserved)
- Per-role model resolution honored
- Token usage round-tripped via `SpawnResult.usage`

**Out of scope.** Multi-driver orchestration (one project running both Codex and Claude Code agents) — defer to v2.

### F2 — Claude Code backend driver (`@swt-labs/claude-code-driver`)

**Scope.** Replace the v1.0 stub with a real driver. 12-event hook taxonomy (PreToolUse / PostToolUse / SessionStart / UserPromptSubmit / etc.). Agent Teams support. Isolation modes.

**Dependencies.** F1 lessons informing the abstraction layer. Claude Code's stable subagent + hook API.

**Complexity.** XL (~6 weeks)

**Success criteria.**
- Same `swt vibe` workflow runs against Claude Code as against Codex
- Hook event taxonomy matches Claude Code's 12 events; SWT's 6 generic events map to a subset
- Agent Teams parallelism works the same as Codex teams (real `team_name` + per-teammate spawn)
- Worktree isolation modes (off / hard / soft) honored

**Out of scope.** Migration helper from existing VBW Claude Code installs (already covered by Phase 11's migration guide).

### F3 — Ollama backend driver (`@swt-labs/ollama-driver`)

**Scope.** Driver wrapping a local Ollama instance. Self-hosted users.

**Dependencies.** F1 (interface stability).

**Complexity.** M (~3 weeks)

**Success criteria.**
- `swt vibe --execute` against a local Ollama model completes the lifecycle
- Model selection respects `model_overrides` config
- Sandbox modes degrade gracefully (Ollama doesn't have native sandboxing — driver wraps in process-level isolation)

**Out of scope.** Hosted Ollama / Ollama Cloud — only the local-process API.

## Tooling

### F4 — Ink TUI dashboard

**Scope.** Per-project live dashboard via `swt watch`. Phase progress, agent status, token spend, remediation round count. Read-only against `.swt-planning/`.

**Dependencies.** None (read-only, no driver coupling).

**Complexity.** M (~2 weeks)

**Success criteria.**
- `swt watch` opens a live TUI scoped to the active milestone
- Updates reflect on-disk state changes within 1s (file-system-watch driven)
- Closes cleanly on Ctrl+C
- Cross-platform: works on Linux/macOS/Windows terminals

See [`.vbw-planning/research/ui-dashboard-tradeoffs.md`](../../.vbw-planning/research/ui-dashboard-tradeoffs.md) for the Ink TUI vs web decision rationale.

### F5 — Plugin marketplace integration

**Scope.** Auto-update via `swt update` when the marketplace is the source of truth. Marketplace listing edits trigger updates without npm publish dance.

**Dependencies.** Codex Plugin Marketplace API exists (today: marketplace is hand-submitted per Phase 12).

**Complexity.** S (~1 week)

**Success criteria.**
- `swt update` queries marketplace API alongside (or instead of) npm registry
- Marketplace-listed version stays in lockstep with npm-published version

### F6 — Auto-derived reference docs

**Scope.** Replace hand-authored CLI/config/artifact reference pages with build-time codegen. The AUTO-DERIVE-CANDIDATE annotations from v1.0 mark the migration targets (`docs/reference/cli.mdx`, `docs/reference/config.mdx`, `docs/reference/artifacts.mdx`).

**Dependencies.** None.

**Complexity.** M (~2 weeks)

**Success criteria.**
- `pnpm docs:gen` produces the three reference pages from source
- Hand-authored "When to override" prose preserved as side-files merged into codegen output (so the human voice survives)
- AUTO-DERIVE-CANDIDATE annotations removed from v1.0 docs once the codegen path is canonical
- Build-time check fails if a Zod schema gains a new key without docs coverage

## Methodology

### F7 — Hook event taxonomy expansion

**Scope.** Add `pre_archive`, `post_phase`, `pre_phase`, `post_uat_fail`, `pre_qa`, `post_qa` hook events alongside the existing `post_archive`.

**Dependencies.** None for the scaffolding; F2 (Claude Code driver) for the load-bearing implementations of the new events.

**Complexity.** S (~1 week)

**Success criteria.**
- Each new event has a documented config key + a sample script
- Existing `post_archive` semantics unchanged
- HookHost narrowing helpers per the abstractions audit (`isPreToolUseEvent(ctx)` style)

### F8 — Telemetry HTTP sender + hosted endpoint

**Scope.** Replace `NoopSender` default with a real HTTP sender wired to a hosted analytics destination (Plausible / PostHog / self-hosted). Configurable cache TTL via `config.json`.

**Dependencies.** Hosted endpoint chosen + procured. (User-side decision — privacy posture matters here.)

**Complexity.** S–M (~1-2 weeks)

**Success criteria.**
- Real HTTP sender behind an `enabled: true` opt-in
- Configurable endpoint via `telemetry.endpoint` config key
- TTL configurable via `telemetry.cache_ttl_hours`
- Privacy contract from v1.0 docs preserved exactly (no PII, anonymous UUIDv4, sanitize-on-send)

## Compatibility commitment

v1.0 → v1.5 has **zero known breaking changes today.** Any breaking change introduced between now and v1.5 cut requires:

1. Announcement in this document under a new `### Breaking change` section
2. Migration script in the v1.5 release
3. Major version bump (1.x → 2.0) if the change actually breaks frontmatter or routing

The methodology contract — phases, plans, summaries, the 11 lifecycle states, the 7-point audit — is treated as semver-permanent. We will not change what `next_phase_state` returns or what `ac_results` records. Adding new states / fields is allowed; renaming or removing existing ones is not.

The CLI surface (`swt init`, `swt vibe`, `swt detect-phase`) is also semver-stable. New flags and modes can be added; existing ones cannot be removed.
