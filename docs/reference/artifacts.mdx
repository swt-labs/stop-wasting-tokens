# Artifacts reference

`@swt-labs/artifacts` exports Zod schemas for every artifact type SWT writes (PLAN.md, SUMMARY.md, VERIFICATION.md, ROADMAP.md, etc.). The schemas listed here are the public surface — they validate every read + write the methodology layer performs.

### `ProjectFrontmatterSchema`

**Notes:** A PROJECT.md frontmatter declares the immutable project metadata (name + description). Set once during `swt init` bootstrap and rarely edited afterwards.

**Fields:**

- `core_value`: `string (optional)`
- `name`: `string (optional)`

### `RoadmapSchema`

**Notes:** A ROADMAP.md frontmatter declares the milestone scope: phases (with names + slugs + completion status) plus the requirement IDs the milestone targets. The methodology layer reads this on every `swt vibe` invocation to compute routing.

**Fields:**

- `phases`: `object[]`
- `project_name`: `string`

### `PlanFrontmatterSchema`

**Notes:** A PLAN.md file's YAML frontmatter declares the contract for a single Lead → Dev handoff. The `must_haves` block is the verification contract: QA reads it back to validate the plan was executed correctly. The `files_modified` array is enforced by the PreToolUse hook — Dev cannot edit files outside this list without amending the plan first.

**Fields:**

- `acceptance_criteria`: `string (optional)`
- `cross_phase_deps`: `string[]`
- `deferred_to_followup`: `string[]`
- `depends_on`: `string[]`
- `effort_override`: `"thorough" | "balanced" | "fast" | "turbo" (optional)`
- `files_modified`: `string[]`
- `forbidden_commands`: `string[]`
- `must_haves`: `string | object[]`
- `phase`: `string`
- `plan`: `string`
- `skills_used`: `string[]`
- `title`: `string`
- `wave`: `number`

### `SummaryFrontmatterSchema`

**Notes:** A SUMMARY.md file's YAML frontmatter is the Dev → QA handoff. The `status` field gates phase routing: `complete` means the plan's must_haves were met; `partial` flags an unfinished plan that needs another Dev pass; `failed` blocks the phase from advancing. The `deviations` array surfaces during QA and gets classified during Round 01 remediation.

**Fields:**

- `ac_results`: `unknown[]`
- `commit_hashes`: `string[]`
- `completed`: `string`
- `deferred_to_followup`: `string[]`
- `deviations`: `unknown[]`
- `files_modified`: `string[]`
- `phase`: `string`
- `plan`: `string`
- `pre_existing_issues`: `string[]`
- `status`: `"complete" | "partial" | "failed" | "in-progress"`
- `tasks_completed`: `number`
- `tasks_total`: `number`
- `title`: `string`

### `ResearchFrontmatterSchema`

**Notes:** A RESEARCH.md frontmatter declares the Scout output for a phase or plan. The `findings` array is the canonical research surface that downstream Lead + Dev agents consume.

**Fields:**

- `files_referenced`: `string[]`
- `findings_summary`: `string`
- `gathered`: `string`
- `live_validation_required`: `boolean`
- `phase`: `string`
- `plan`: `string (optional)`
- `sources_consulted`: `string[]`

### `RemediationPlanFrontmatterSchema`

**Notes:** An R{NN}-PLAN.md frontmatter for a QA remediation round. The `fail_classifications` array (one entry per FAIL row in the source verification) is the contract the deterministic gate validates — every entry must classify as code-fix / plan-amendment / process-exception.

**Fields:**

- `fail_classifications`: `object[]`
- `known_issue_resolutions`: `string[]`
- `known_issues_input`: `string[]`
- `phase`: `string`
- `round`: `string`
- `tasks_total`: `number`
- `title`: `string`

### `RemediationSummaryFrontmatterSchema`

**Notes:** An R{NN}-SUMMARY.md frontmatter for a QA remediation round. Mirrors the SUMMARY.md shape (status, tasks_completed, files_modified, deviations) but is scoped to a remediation round rather than an original plan.

**Fields:**

- `commit_hashes`: `string[]`
- `completed`: `string`
- `deviations`: `unknown[]`
- `files_modified`: `string[]`
- `known_issue_outcomes`: `string[]`
- `phase`: `string`
- `round`: `string`
- `status`: `"complete" | "partial" | "failed" | "in-progress"`
- `tasks_completed`: `number`
- `tasks_total`: `number`
- `title`: `string`
