# Migration from VBW

[Vibe Better with Claude Code](https://github.com/Hesreallyhim/awesome-claude-code) (VBW) is the methodology SWT is based on. By design, **SWT reads VBW's artifacts unchanged** — same frontmatter shapes, same lifecycle states, same routing logic. The v3 runtime swap (Codex CLI → Pi) did not change the methodology contract; VBW projects continue to round-trip through SWT v3.0.

If you have a working VBW project, migrating to SWT v3 is `mv .vbw-planning .swt-planning` plus a few config tweaks. Parity evidence: `.vbw-planning/phases/05-agent-tuning-parity-testing/PARITY-REPORT.md` records the 7-agent output-equivalence harness against a known-good fixture phase.

## Frontmatter compatibility

Every VBW frontmatter shape parses cleanly through SWT's Zod schemas. The transforms in `@swt-labs/shared/schemas/` accept both shapes:

| Artifact                 | VBW shape                                         | SWT shape                                          | Migration               |
| ------------------------ | ------------------------------------------------- | -------------------------------------------------- | ----------------------- |
| `PLAN.md`                | `must_haves: [string]` or `[MustHaveBlock]`       | Same                                               | None                    |
| `SUMMARY.md`             | `ac_results: [{id, must_have, status, evidence}]` | `ac_results: [{id, criterion, verdict, evidence}]` | Auto-normalized on read |
| `VERIFICATION.md`        | Markdown table + frontmatter                      | Same shape, Zod-validated                          | None                    |
| `UAT.md`                 | YAML + body                                       | Same shape + optional `severity_counts`            | None                    |
| `RESEARCH.md`            | Markdown body                                     | Same                                               | None                    |
| `CONTEXT.md`             | YAML + 3-section body                             | Same                                               | None                    |
| `MILESTONE-CONTEXT.md`   | YAML + 6-section body                             | Same                                               | None                    |
| `REMEDIATION-PLAN.md`    | `fail_classifications[]` array                    | Same                                               | None                    |
| `REMEDIATION-SUMMARY.md` | `known_issue_outcomes[]` array                    | Same                                               | None                    |
| `DEBUG-SESSION.md`       | `agent` + structured body                         | Same                                               | None                    |

The "Auto-normalized" entry for SUMMARY means: a VBW SUMMARY with `must_have` and `status` keys is parsed and re-rendered as `criterion` and `verdict` on save. The original semantics are preserved.

## Lifecycle state parity

The 11 SWT lifecycle states map 1:1 to VBW's. Same names, same triggers, same routing decisions. `swt detect-phase --bash-format` produces VBW-compatible `key=value` output — VBW shell scripts work against SWT's planning dir without modification.

## Config compatibility

VBW's `config.json` keys round-trip cleanly through SWT v3. Reading a VBW config through SWT works; SWT v3 silently ignores keys not yet in the v3 schema (e.g., VBW-specific `agent_max_turns.architect_v2`).

The reverse is true within v3: VBW reads an SWT v3 config without complaining about additive keys (`schema_version`, `router_strategy`, per-role `tier`).

## What's different

The actual differences are operational, not methodological:

- **Distribution.** VBW is a Claude Code plugin (bash + commands). SWT is an npm package (TypeScript CLI).
- **Backend.** VBW runs in Claude Code only. SWT v3 runs against [`@earendil-works/pi-coding-agent`](https://www.npmjs.com/package/@earendil-works/pi-coding-agent) which natively supports Anthropic, OpenAI, OpenRouter, Google, Bedrock, and local Ollama.
- **Dashboard.** SWT v3 ships a Hono + Solid + SSE web dashboard as the primary surface (TDD3 §15). Bare `swt` (no args) launches it. VBW had no web surface.
- **Worktree isolation.** SWT v3 dispatches parallel Dev tasks across per-task git worktrees (per ADR-008). Default `worktree_isolation: 'off'` for v3.0; flip to `'auto'` opt-in. VBW serialised parallel work.

Methodologically: identical.

## When to migrate

Migrate from VBW to SWT v3 if:

- You want vendor-neutral runtime support (Anthropic / OpenAI / OpenRouter / Google / Bedrock / Ollama) without rewiring per-provider.
- You want a single CLI distributed via npm rather than a Claude Code plugin install.
- You want SWT's stricter Zod-validated artifact schemas (e.g., parse-on-read catches malformed YAML earlier than VBW's freeform parser).
- You want worktree-isolated parallel Dev dispatch + Budget Gate + cassette-deterministic CI.

Stay on VBW if:

- Your team is committed to Claude Code as the only LLM surface AND you do not need the v3 dashboard / worktree / budget primitives.
- You rely on VBW-specific features that SWT v3 has not yet wired (Agent Teams beyond per-task workers).
- You want plugin-marketplace install + auto-update through Claude Code.

Both projects share the methodology. You can run them side-by-side during evaluation — they don't conflict at the file level (`.vbw-planning/` vs `.swt-planning/`).
