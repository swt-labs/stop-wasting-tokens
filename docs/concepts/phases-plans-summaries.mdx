# Phases, plans, summaries

SWT projects produce many files. They all have the same purpose: capture a contract that the runtime + future agents can read deterministically.

## Disk layout

```
.swt-planning/
  PROJECT.md              # name + core purpose
  REQUIREMENTS.md         # uncovered → covered as phases ship
  ROADMAP.md              # phase decomposition
  STATE.md                # current phase, decisions, todos, blockers
  CONTEXT.md              # milestone-level decisions (optional)
  config.json
  conventions.json
  phases/
    01-storage-layer/
      01-CONTEXT.md       # phase-level discussion outputs (optional)
      01-RESEARCH.md      # phase-wide research from Scout (optional)
      01-01-PLAN.md       # plan 1
      01-01-SUMMARY.md    # outcome of plan 1
      01-02-PLAN.md
      01-02-SUMMARY.md
      01-VERIFICATION.md  # QA verdict
      01-UAT.md           # interactive UAT outcome
      remediation/        # round-scoped remediation artifacts (if needed)
        uat/
          round-01/
            R01-PLAN.md
            R01-SUMMARY.md
            R01-VERIFICATION.md
            R01-UAT.md
  milestones/             # archived shipped milestones
    01-todo-cli/
      ROADMAP.md
      STATE.md
      phases/...
      SHIPPED.md
```

## Artifact kinds

Every artifact has typed Zod-validated frontmatter. The runtime reads via `@swt-labs/artifacts/schemas/`.

| Kind                     | Path                                               | Purpose                                        |
| ------------------------ | -------------------------------------------------- | ---------------------------------------------- |
| **PROJECT**              | `.swt-planning/PROJECT.md`                         | Name + core purpose                            |
| **REQUIREMENTS**         | `.swt-planning/REQUIREMENTS.md`                    | Hierarchical list of REQ-IDs                   |
| **ROADMAP**              | `.swt-planning/ROADMAP.md`                         | Phase decomposition + dependency table         |
| **STATE**                | `.swt-planning/STATE.md`                           | Current phase, decisions, activity log         |
| **CONTEXT**              | `.swt-planning/CONTEXT.md` (milestone-level)       | Decomposition rationale + scope coverage       |
| **PHASE-CONTEXT**        | `phases/NN-slug/NN-CONTEXT.md`                     | Per-phase discussion engine output             |
| **RESEARCH**             | `phases/NN-slug/NN-RESEARCH.md`                    | Phase-wide research findings (Scout output)    |
| **PLAN**                 | `phases/NN-slug/NN-MM-PLAN.md`                     | Tight ordered task list with must-haves        |
| **SUMMARY**              | `phases/NN-slug/NN-MM-SUMMARY.md`                  | Outcome — must-have results, deviations, files |
| **VERIFICATION**         | `phases/NN-slug/NN-VERIFICATION.md`                | QA verdict against the phase contract          |
| **UAT**                  | `phases/NN-slug/NN-UAT.md`                         | Interactive UAT outcome (PASS / issues_found)  |
| **DEBUG-SESSION**        | `phases/NN-slug/debug/SS-DEBUG.md`                 | Investigation/findings/resolution log          |
| **REMEDIATION-PLAN**     | `phases/NN-slug/remediation/.../R{RR}-PLAN.md`     | Per-round remediation task list                |
| **REMEDIATION-SUMMARY**  | `phases/NN-slug/remediation/.../R{RR}-SUMMARY.md`  | Per-round remediation outcome                  |
| **REMEDIATION-RESEARCH** | `phases/NN-slug/remediation/.../R{RR}-RESEARCH.md` | Per-round Scout research                       |

See the full schema reference in [Artifact schemas](/reference/artifacts).

## Why frontmatter, not JSON?

Frontmatter is human-readable, git-diff-friendly, and lets you edit any artifact in any text editor without breaking the runtime. The body of each file is for humans (the narrative); the frontmatter is for the runtime (the contract).

Every parser uses Zod transforms to handle backwards-compatibility. A VBW-shipped `PLAN.md` parses cleanly through `@swt-labs/artifacts` — no migration script needed.

## Phase, plan, summary cardinality

- **Per phase:** zero or one `CONTEXT.md`, zero or one `RESEARCH.md`, one `VERIFICATION.md` (after QA), zero or one `UAT.md` (after Verify mode).
- **Per phase:** one or more `PLAN.md` files. Each one has exactly one matching `SUMMARY.md` after execution.
- **Per remediation round:** one `R{RR}-PLAN.md`, one `R{RR}-SUMMARY.md`, one `R{RR}-VERIFICATION.md`, one `R{RR}-UAT.md`, optionally one `R{RR}-RESEARCH.md`.

## What lives where

- **Decisions that survive the milestone** → `STATE.md` (Key Decisions table, Codebase Profile)
- **Decisions scoped to one milestone** → `.swt-planning/CONTEXT.md`
- **Decisions scoped to one phase** → `NN-CONTEXT.md`
- **Outcome of a single plan** → `NN-MM-SUMMARY.md`

The runtime knows how to load each scope level on demand via `compile-context.sh`. You don't have to think about it day-to-day — but knowing the cardinality helps when you're inspecting state or fixing something by hand.
