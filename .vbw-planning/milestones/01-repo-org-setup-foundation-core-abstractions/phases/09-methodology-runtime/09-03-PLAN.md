# Phase 9 / Plan 03: Bootstrap + Scope mode handlers

## Why third

PLAN 02 wired the routing layer; every mode is a stub right now. This plan replaces the **input-side** stubs (Bootstrap = greenfield project setup; Scope = milestone setup with N phases). After this lands, a fresh `swt vibe` from an empty directory can drive the project to a planned-and-decomposed state without a discussion engine.

## Tasks

### T1 — discovery.ts
Zod schema for `discovery.json`: `{ answered: string[], inferred: { text: string; priority: 'must-have'|'nice-to-have'|'out-of-scope' }[], deferred: string[] }` plus `readDiscovery(planningDir)` and `writeDiscovery(planningDir, data)`. Atomic write via writeAtomically.

### T2 — project.ts
`writeProject(opts)` accepts `{ planningDir, name, description, core_value? }` and emits a `PROJECT.md` shaped like the VBW template (h1, description paragraph, `**Core value:**` line, Requirements section with Validated/Active/Out-of-scope subsections, Constraints section, Key Decisions table). Idempotent — safe to call on an existing file.

### T3 — requirements.ts
`writeRequirements(opts)` reads `discovery.json`, generates REQ-IDs (`REQ-01`..`REQ-NN`), groups by priority, and writes `REQUIREMENTS.md` in the v1 shape (Domain Context, v1 Requirements by category, v2 Requirements, Out of Scope). Uses `formatFrontmatter` for YAML headers where the VBW template emits frontmatter.

### T4 — roadmap.ts
`writeRoadmap(opts)` accepts `{ planningDir, project_name, phases: PhaseEntry[] }` and emits `ROADMAP.md` matching the body shape we wrote manually in Phase 1: progress table, phase list, per-phase sections with goal / requirements / success criteria / dependencies. Numbered headings 1..N, IDs match the phase position pad.

### T5 — state.ts
`writeState(opts)` accepts `{ planningDir, project_name, milestone_name, phase_count }` and emits a fresh `STATE.md` matching the shape we use today (Project, Milestone, Current Phase, Phase Status, Key Decisions, Todos, Blockers, Activity Log).

### T6 — claude.ts
`writeOrUpdateClaudeMd(opts)` accepts `{ planningDir, project_name, core_value, existing_path? }`. If `existing_path` is given and exists, refresh only the canonical VBW-owned sections (`## Active Context`, `## VBW Rules`, `## Plugin Isolation`, `## Code Intelligence`) and add `## Code Intelligence` only when no equivalent guidance is present anywhere in the file. Otherwise produce a new CLAUDE.md.

### T7 — context.ts
`writePhaseContext(opts)` writes `<planningDir>/phases/<NN>-<slug>/<NN>-CONTEXT.md` with frontmatter (`phase`, `plan`, `gathered`, `pre_seeded`) + Goal / Notes sections.
`writeMilestoneContext(opts)` writes `<planningDir>/CONTEXT.md` with the milestone-level shape (Scope Boundary, Decomposition Decisions, Requirement Mapping, Key Decisions, Deferred Ideas).

### T8 — Bootstrap handler
Spec: registry replacement for `kind='bootstrap'`. Handler steps:
1. Read CLI args from the route's metadata (Bootstrap accepts `project_name`, `description`, `core_value` via flags or as a tail of positionals).
2. If `project_name` and `description` are missing, return `NotImplementedError(kind='bootstrap', roadmap_pointer='Phase 9 / Plan 03b — interactive discussion engine')`. (Args path works today; interactive path lands in 03b.)
3. Call `writeProject`, `writeDiscovery` (empty if absent), `writeRequirements`, `writeRoadmap` (empty array — phases come later via `swt vibe scope` or PLAN 04), `writeState`, `writeOrUpdateClaudeMd`.
4. Return `HandlerResult { exit: 0, ranTo: 'completion' }`.

### T9 — Scope handler
Spec: registry replacement for `kind='scope'`. Handler steps:
1. Read phases payload from `<planningDir>/phases.json` (created by user or future tooling). If absent, return `NotImplementedError(kind='scope', roadmap_pointer='Phase 9 / Plan 03b — interactive scope decomposition')`.
2. Validate against `RoadmapSchema.phases`.
3. For each phase: `createPhaseDir` (already in artifacts).
4. Call `writeRoadmap`, `writeState` (with phase_count = phases.length), `writeMilestoneContext`.
5. Return `HandlerResult { exit: 0, ranTo: 'completion' }`.

### T10 — Wire into vibe registry
Update `vibe/handlers/stubs.ts`'s `buildStubRegistry` to accept an optional override map, OR expose a separate `buildRegistry` that registers real handlers + falls back to stubs for the rest. Simpler approach: export a `realHandlers` array and let the CLI assemble `registry = buildStubRegistry().override(realHandlers)`.

### T11 — Vitest
Per-writer round-trip on temp dirs (10 of them) + handler integration tests that run Bootstrap + Scope end-to-end against a temp `.swt-planning/` and assert the output trees.
