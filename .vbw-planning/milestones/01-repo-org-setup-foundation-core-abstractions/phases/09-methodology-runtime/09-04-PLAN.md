# Phase 9 / Plan 04: Plan + Execute mode handlers

## Why fourth

PLAN 03 ships the input side (Bootstrap + Scope) — it produces planning artifacts from a JSON payload. PLAN 04 ships the build side: given a planned phase, drive Dev + Wave orchestration to completion and persist SUMMARY.md per plan. After this lands, `swt vibe` on a freshly scoped project can ride from Phase 1 → all_done (modulo verification, which is PLAN 06).

## Tasks

### T1 — orchestration/plan-input.ts
`resolvePlanInput({phaseDir, planningDir})` reads:
- `<phaseDir>/<NN>-CONTEXT.md` (if present)
- `<phaseDir>/<NN>-RESEARCH.md` (if present)
- `<planningDir>/REQUIREMENTS.md`
- `<phaseDir>/<NN>-NN-PLAN.md` for each existing plan (so re-runs are idempotent)
Returns a typed `PlanInput` with goal, must-haves, requirement-mapping, and the list of plan ids already present.

### T2 — handlers/plan.ts
`planHandler` reads PlanInput, spawns the Lead via the AgentSpawner (or, when none is available, a deterministic in-process LeadStrategy that produces N=1 plan from the must-haves), persists `<phaseDir>/<NN>-NN-PLAN.md` for each plan, returns HandlerResult.

### T3 — orchestration/waves.ts
`groupByWave(plans)` returns `Wave[]`, each `Wave` holds plans whose `wave` field matches and `depends_on` references plans in earlier waves. `validateDisjointFiles(wave)` throws `RoutingError` when two same-wave plans declare overlapping `files_modified`.

### T4 — orchestration/dev-runner.ts
`runDev({plan, agentSpawner, io})` issues a `SpawnRequest` with the `dev` AgentSpec, captures the structured `SpawnResult.handoff` (Dev summary), and returns the parsed `DevSummaryPayload` + `text`.

### T5 — orchestration/summary-writer.ts
`writeSummary({phaseDir, plan, devSummary})` formats a SUMMARY.md with frontmatter (`phase`, `plan`, `title`, `status`, `tasks_completed`, `tasks_total`, `commit_hashes`, `files_modified`, `deviations`) plus a body section per task.

### T6 — handlers/execute.ts
`executeHandler` reads all `<phaseDir>/<NN>-NN-PLAN.md`, groups by wave, validates disjoint files, then for each wave runs every plan (in parallel within the wave, sequential across waves) via `runDev` + `writeSummary`. Returns HandlerResult{exit:0}.

### T7 — handlers/plan-and-execute.ts
Chain handler: call planHandler then executeHandler. Pass Dev-side state from plan to execute via the in-memory PlanInput so the second pass doesn't re-read disk.

### T8 — Wire into vibe registry
Update CLI `vibe.ts` to register `[bootstrapHandler(), scopeHandler(), planHandler({...}), executeHandler({...}), planAndExecuteHandler({...})]` via `buildVibeRegistry`.

### T9 — Vitest
- `plan.test.ts`: PlanInput resolution + handler emits N plans with the right shape, plus idempotence on a phase that already has the right number of PLAN.md files.
- `execute.test.ts`: wave grouping correctness, disjoint-files invariant, mock Dev runs end-to-end against a temp dir using `@swt-labs/core/test/mock-driver`, SUMMARY.md round-trip.
- `plan-and-execute.test.ts`: chained handler runs full pipeline against a temp dir.
