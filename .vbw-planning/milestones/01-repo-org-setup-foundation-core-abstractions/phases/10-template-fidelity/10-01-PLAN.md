# Phase 10 / Plan 01: Upgrade PLAN + SUMMARY schemas to VBW-grade

## Why first

Phase 10 brings SWT's planning artifacts up to VBW shape. Plan 01 picks the two highest-value documents — PLAN.md and SUMMARY.md — because every other VBW artifact (VERIFICATION/UAT/REMEDIATION-*/CONTEXT/MILESTONE-CONTEXT) layers on top of these two. After this lands, any VBW-shipped PLAN/SUMMARY parses cleanly through SWT, the 7-point audit (PLAN 09-07) can read VBW-grade `ac_results`, and downstream plans can extend the schema set incrementally.

## Layout

```
packages/artifacts/src/schemas/plan.ts                 # PlanFrontmatterSchema + helpers
packages/artifacts/src/schemas/summary.ts              # SummaryFrontmatterSchema + helpers
packages/artifacts/src/schemas/index.ts                # extend the barrel
packages/artifacts/src/frontmatter.ts                  # parser/formatter accepts inline JSON-array-of-objects
packages/artifacts/test/schemas/plan.test.ts
packages/artifacts/test/schemas/summary.test.ts
packages/artifacts/test/fixtures/vbw-summary-sample.md
packages/artifacts/test/fixtures/vbw-plan-sample.md
```

## Tasks

### T1 — frontmatter parser/formatter extension
Extend `formatScalarYaml` so values that are arrays of objects render as
`key: [{"...": "..."}, ...]` (single-line JSON). Extend `parseScalarYaml`
to recognise `key: [{ ... }, ... ]` and parse as JSON via `JSON.parse` when
the inline array opens with `{`. Existing `[item, item]` string-array
parsing still wins when the array does NOT contain `{`.

### T2 — schemas/plan.ts
- `MustHaveBlockSchema = z.object({ truths: z.array(z.string()).default([]), artifacts: z.array(z.string()).default([]), key_links: z.array(z.string()).default([]) })`
- `MustHaveSchema = z.union([z.string(), MustHaveBlockSchema])`
- `PlanFrontmatterSchema` with all the VBW fields, every optional `cross_phase_deps[]`, `effort_override`, `forbidden_commands[]`, `skills_used[]`, `files_modified[]`. Required: phase (`/^\d{2}$/`), plan (`/^\d{2}[a-z]?$/`), title, wave (positive int), depends_on (string array), must_haves (array of MustHaveSchema, min 1).
- `readPlanFrontmatter(raw)` and `writePlanFrontmatter(fm, body)` helpers.

### T3 — schemas/summary.ts
- `AcResultSchema = z.object({ id, criterion (alias must_have), verdict (alias status), evidence }).transform(...)` — accepts the VBW-grade shape OR the SWT-1.0 shape and normalizes to `{id, criterion, verdict, evidence}`.
- `DeviationSchema = z.object({id, type?, description, rationale? | resolution?})` — accepts both VBW and SWT shapes; normalizes to `{id, type?, description, resolution?}`.
- `SummaryFrontmatterSchema` with phase/plan/title/status/completed/tasks_completed/tasks_total/ac_results/pre_existing_issues/commit_hashes/files_modified/deviations.
- `readSummaryFrontmatter(raw)` and `writeSummaryFrontmatter(fm, body)` helpers.

### T4 — schemas/index.ts
Extend the barrel: re-export PLAN + SUMMARY schemas alongside roadmap/state/project/requirements.

### T5 — Test fixtures
- `vbw-summary-sample.md`: a verbatim copy of a small VBW-grade SUMMARY.md with `ac_results: [{id, must_have, status, evidence}]` and structured deviations.
- `vbw-plan-sample.md`: a small VBW-grade PLAN.md with structured `must_haves` (mixed strings + blocks).

### T6 — Vitest
- `plan.test.ts`: parse string-only must_haves; parse structured must_haves blocks; round-trip both; reject malformed shapes (wrong types, missing required fields).
- `summary.test.ts`: parse VBW-grade ac_results; parse SWT-1.0 ac_results (status instead of verdict); normalize both into the Plan-10 shape; round-trip; parse the vbw-summary-sample fixture without data loss.

### T7 — Commit + summary
`feat(artifacts): VBW-grade PLAN + SUMMARY schemas with backwards compat (Phase 10 / PLAN 01)`.
