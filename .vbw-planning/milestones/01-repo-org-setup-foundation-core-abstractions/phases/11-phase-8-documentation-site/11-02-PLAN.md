# Phase 11 / Plan 02: Reference + recipes + migration guide + v1.5 roadmap

## Why second

Plan 02 fills the 4 sections Plan 01 left empty (`reference/`, `recipes/`, `migration/`, `v1-5-roadmap/`). The split exists because Plan 01's deliverable — a working Mintlify scaffold with concepts content — is independently shippable; Plan 02 is content-heavy and benefits from being authored against the live scaffold (preview-driven authoring).

## Tasks

### T1 — reference/cli.mdx
For each command exported by `packages/cli/src/main.ts`, render a section with:
- Synopsis (`swt <cmd> [flags]`)
- Description
- Flags table (flag / type / default / description)
- 2-3 worked examples with expected output
- "See also" cross-links

Source the command list by reading `packages/cli/src/main.ts` and the per-command files under `packages/cli/src/commands/`. Where descriptions are sparse in source, hand-author. Add an annotation comment at the top of each section: `<!-- AUTO-DERIVE-CANDIDATE: keep in sync with packages/cli/src/commands/<file>.ts -->` so future v1.5 auto-gen can find them.

### T2 — reference/config.mdx
Read `packages/core/src/config/defaults.json` and `packages/core/src/config/schema.ts`. For each key, render:
- Key name
- Type (TypeScript)
- Default
- Description (from Zod `.describe()` calls; hand-author if missing)
- "When to override" note (1-2 sentences)
- Example value

Group by category: Lifecycle (effort/autonomy/auto_commit), Routing (planning_tracking/auto_push/verification_tier), Agents (prefer_teams/max_tasks_per_plan/agent_max_turns/qa_skip_agents/model_profile/model_overrides), Methodology (context_compiler/require_phase_discussion/auto_uat/max_uat_remediation_rounds), Display (visual_format/plain_summary/statusline_*).

### T3 — reference/artifacts.mdx
For each schema file in `packages/artifacts/src/schemas/`, render:
- Schema name
- Purpose (1 sentence)
- Frontmatter fields table (field / type / required? / description)
- A real example block (mark as ```yaml or ```markdown)
- Link to the source file (e.g., `packages/artifacts/src/schemas/plan.ts`)
- Round-trip helper signatures (e.g., `readPlanFrontmatter(text: string): PlanDoc`)

Cover all 12: PlanFrontmatterSchema, SummaryFrontmatterSchema, VerificationDocSchema, UatDocSchema, ResearchFrontmatterSchema, StandaloneResearchFrontmatterSchema, RemediationPlanFrontmatterSchema, RemediationResearchFrontmatterSchema, RemediationSummaryFrontmatterSchema, DebugSessionSchema, PhaseContextSchema, MilestoneContextSchema.

### T4 — recipes
Author 5 walkthrough pages (~400-600 words each):

**`recipes/greenfield.mdx`:** Brand-new project. `mkdir my-app && cd my-app && swt init`. Walk through bootstrap (project name, requirements via discussion engine), scope into 3-4 phases, plan + execute Phase 1, verify, archive. End-to-end flow with command outputs.

**`recipes/brownfield.mdx`:** Existing codebase. `cd existing-repo && swt init` (codebase mapping kicks in), bootstrap with `discussion_mode=assumptions`, scope around brownfield concerns, run a remediation-style first phase. Highlight the codebase profile in STATE.md.

**`recipes/gsd-migration.mdx`:** Repo with `.planning/` from GSD. `swt init` detects + offers archive import. Walk through choosing "Import" vs "Skip", review `.swt-planning/gsd-archive/INDEX.json`, plan first SWT-native phase referencing GSD context.

**`recipes/uat-remediation.mdx`:** Phase has UAT issues. `swt vibe` routes to remediation. Walk through round 01: research → plan → execute → re-verify. Show a recurring-issue annotation in round 02 ("⚠ RECURRING (failed in 2 recorded rounds)").

**`recipes/custom-hooks.mdx`:** Configure `hooks.post_archive` to run a custom Slack/Discord webhook after archive. Show the `config.json` shape and a working `bash` script under `.swt-planning/hooks/post-archive.sh`.

### T5 — migration + v1.5 roadmap
**`migration/from-vbw.mdx`:** Feature parity table (Bootstrap / Scope / Plan / Execute / Verify / Archive / Discuss / Add-Insert-Remove / UAT remediation / QA remediation / Audit gate / Hook event taxonomy / Discussion engine — all ✅ except real-Codex AgentSpawner which is ⚠ blocked). Note that frontmatter shapes are wire-compatible (Phase 10 schemas accept VBW's PLAN/SUMMARY/VERIFICATION/UAT/etc. via Zod transforms).

**`migration/step-by-step.mdx`:** Concrete steps:
1. `cp -r .vbw-planning .swt-planning` (or rename)
2. Validate config: `swt detect-phase --json` should round-trip
3. Update repo `.gitignore` if needed
4. Update CI to call `swt` instead of `vbw` commands
5. Test with `swt vibe` on a known-good phase

**`migration/breaking-changes.mdx`:** "v1.0 has zero breaking changes vs VBW. Frontmatter shapes are forward-compatible via Zod transforms. v1.5 will introduce {placeholder: TBD when v1.5 milestone scopes}." 1-page placeholder.

**`v1-5-roadmap/index.mdx`:** Roadmap doc listing planned v1.5 deliverables: real Codex AgentSpawner wiring (unblocks live execute), Claude Code backend driver, Ollama backend driver, Ink TUI dashboard, plugin marketplace listing, auto-derived reference docs (replace hand-authored CLI/config/artifact refs with codegen). Each item: 1-paragraph rationale + status (planned / in-design / blocked).

Update `apps/docs/docs.json` to point each navigation group at the real `.mdx` files. Re-run the structure vitest from PLAN 11-01 — must pass.
