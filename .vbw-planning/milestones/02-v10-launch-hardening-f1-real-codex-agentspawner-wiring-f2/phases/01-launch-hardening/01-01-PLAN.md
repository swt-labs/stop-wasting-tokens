<objective>
Close audit C1, C2, and C5: stop emitting CLAUDE.md by default, stop stamping `## VBW Rules` into project-doc bodies, and stop leaking `.vbw-planning/` in CLI help text. After this plan ships, a fresh `swt init` produces an AGENTS.md whose SWT-managed block is correctly branded — and legacy CLAUDE.md projects migrate cleanly without losing user-authored content.
</objective>
<context>
Codex CLI consumes AGENTS.md as the project-doc convention; CLAUDE.md is Claude Code's convention. SWT's v1.0 backend is Codex-only (per PROJECT.md), so the bootstrap path must produce AGENTS.md. The existing `writeAgentsMdBlock` in `@swt-labs/codex-driver` correctly uses fenced `<!-- SWT BEGIN -->` / `<!-- SWT END -->` markers (so SWT coexists with other tools that may also write to AGENTS.md, e.g. Cursor, Cline, Aider) — but `bootstrap.ts` doesn't call it. `writeOrUpdateClaudeMd` in `@swt-labs/artifacts` uses VBW-style `## Section` markers and stamps a literal `## VBW Rules` heading. The migration shim in this plan recognises legacy `VBW Rules` headings during refresh and rewrites them to `SWT Rules` so existing SWT/VBW projects coexist with the v1.0 cleanup.

CLAUDE.md generation does not get deleted — it's retained as an opt-in path for the future Claude Code backend (v1.5 F2) and for legacy SWT projects that already have a CLAUDE.md.
</context>
