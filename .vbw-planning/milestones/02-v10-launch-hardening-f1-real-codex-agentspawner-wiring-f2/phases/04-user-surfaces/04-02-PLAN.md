<objective>
Extend `swt update` to query the Codex Plugin Marketplace alongside the npm registry, surfacing any version divergence between the two sources. Ship structural support today; the published default leaves marketplace lookup dormant until users opt in via `config.marketplace.endpoint` (the public Codex Plugin Marketplace API isn't documented as a stable surface yet).
</objective>
<context>
**Decision resolved 2026-05-07 (user via AskUserQuestion before Phase 04 planning):**
- Marketplace strategy = wait-on-registration; ship structural support. Parameterized endpoint via config; default null. When the real marketplace API goes public, users set `marketplace.endpoint` and the code path activates without requiring a new release.

**Marketplace-registry helper signature** (in `packages/cli/src/lib/marketplace-registry.ts`):
```typescript
export interface MarketplaceQueryOptions {
  readonly endpoint: string;
  readonly packageName: string;
  readonly fetchImpl?: typeof fetch;
  readonly noCache?: boolean;
  readonly cachePath?: string;
  readonly now?: () => number;
  readonly cacheTtlHours?: number;
}

export interface MarketplaceVersion {
  readonly version: string;
  readonly source: 'marketplace';
  readonly fetchedAt: number;
  readonly fromCache: boolean;
}

export async function queryMarketplaceVersion(
  opts: MarketplaceQueryOptions,
): Promise<MarketplaceVersion>;
```

The helper mirrors `queryLatestVersion` from `lib/npm-registry.ts` (same cache shape via `cachePath`, same `now()` injection for tests, same fetchImpl seam). The expected response is a JSON document with `{name, latest_version, ...}` shape — exact schema TBD when the marketplace publishes its API; the helper validates via Zod with safe-parse and surfaces any schema drift as a clear error.

**Updated `swt update` flow:**
- Existing path: query npm registry → display latest npm version + upgrade commands.
- New path (when `config.marketplace.endpoint` is set): query both registries in parallel via `Promise.allSettled`, display both versions, flag divergence with a `⚠ Marketplace version (X.Y.Z) differs from npm (A.B.C)` line.
- New CLI flag: `--no-marketplace` skips the marketplace path even when configured.
- Failure handling: marketplace lookup failure (non-2xx, network error, schema-drift) is logged at debug level, never blocks the npm path — the F5 success criterion specifies "alongside (or instead of)" not "replacing".

**No real endpoint URL hardcoded.** The published `defaults.json` ships with no `marketplace` block; the default behavior is exactly today's npm-only path. Tests inject a stub endpoint URL.

Create `packages/cli/src/lib/marketplace-registry.ts` paralleling `lib/npm-registry.ts`:
- Export `MarketplaceQueryOptions` + `MarketplaceVersion` types
- Export `queryMarketplaceVersion(opts)` async function:
  1. Resolve effective fetchImpl (default globalThis.fetch), cachePath (default `~/.cache/swt/marketplace.json`), now (default Date.now), cacheTtlHours (default 24).
  2. Cache check: if `noCache !== true` and cached entry for `endpoint+packageName` is within `cacheTtlHours`, return `{...entry, fromCache: true}`.
  3. Cache miss: GET `${endpoint}/${packageName}` (URL composition via `new URL`). 5-second AbortSignal timeout.
  4. Parse JSON; validate via a minimal Zod schema (`{latest_version: z.string()}`).
  5. Persist cache, return `{version, source: 'marketplace', fetchedAt: now(), fromCache: false}`.
  6. Errors: throw a typed `MarketplaceQueryError` with cause chain (caller logs + falls back to npm path).
