After Plan 02-03 ships, `swt vibe --execute` against a real project actually spawns Codex subagents — the inline-orchestrator pattern that Phase 01 used (per the v1.0 SHIPPED.md note) becomes the legacy fallback.
</objective>
<context>
**Decisions resolved 2026-05-07 (user via AskUserQuestion):**
1. **TOML parser** = `@iarna/toml` (added as a runtime dep on `@swt-labs/methodology`).
2. **Templates location** = move `agents-templates/` to `packages/methodology/templates/agents/`. Resolver locates them via `new URL('../../templates/agents/', import.meta.url)`. This makes the templates reachable in both the monorepo and once published — npm includes everything not in `.npmignore`.
3. **Install timing** = lazy install on first spawn per role, implemented via a new `LazyInstallSpawner` wrapper in methodology orchestration. The wrapper holds a base `AgentSpawner` and a `resolveSpec(role)` callback; on `spawn(request)` it calls `base.installAgent(spec)` once per role before delegating. `cleanup()` calls `base.removeAgent(role)` for each installed role. The CLI vibe handler builds the wrapper around `CodexAgentSpawner`.

The executeHandler at packages/methodology/src/vibe/handlers/execute.ts already accepts `opts.spawner` and `opts.devSpec`; when undefined it throws NotImplementedError. Plan 02-03 closes that door from the CLI side.

**Model resolution policy** (codified in agent-spec-resolver.ts):
1. If `config.model_overrides[role]` is set, use it
2. Else read `agents-templates/{role}.toml` from the shipped package; use its `model` field
3. Else fall back to a backend-default sentinel (e.g. `"default"` — Codex CLI handles this)

The `agents-templates/` directory ships INSIDE the @swt-labs/methodology (or @swt-labs/codex-driver) package — not the consumer's repo. v1.0 placed them at the monorepo root; v1.5 must decide which package owns them. **Plan 02-03 deviation candidate**: the templates' canonical location may need to move to a package's `templates/` dir to be reachable in npm-installed scenarios; if so, document the move in the SUMMARY.

**Per-role spec composition**: resolveAgentSpec merges:
- role (from input)
- model (from precedence rules above)
- reasoning_effort (from agents-templates {role}.toml `model_reasoning_effort` field)
- developer_instructions (from agents-templates {role}.toml `developer_instructions` field)
- allowed_mcp_servers (from agents-templates {role}.toml + config.json mcp_overrides if present)
- sandbox_mode (from agents-templates {role}.toml)
- max_turns (from agents-templates {role}.toml + config.json agent_max_turns[role] override)

config.json's `agent_max_turns` and `model_overrides` already exist (v1.0 ship) — this plan just reads them.

**Shutdown**: when executeHandler completes (or errors), call `spawner.removeAgent(role)` for each installed role. Add a `Promise.allSettled` cleanup at the end of the run loop in executeHandler if not already present.
</context>
