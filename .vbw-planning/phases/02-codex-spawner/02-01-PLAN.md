<objective>
Concretise the AgentSpawner abstraction with a real Codex backend implementation. CodexAgentSpawner is the class methodology's execute handler will inject when v1.5 wires up real spawning. v1.0 already shipped the low-level `spawnCodex` function and `emitAgentToml`; F1 merely fuses them into a class that satisfies the AgentSpawner interface.
</objective>
<context>
The AgentSpawner contract at packages/core/src/abstractions/AgentSpawner.ts has three methods: `installAgent(spec)`, `spawn(request)`, `removeAgent(role)`. The Codex backend already has the building blocks:

- `emitAgentToml(spec)` in packages/codex-driver/src/toml/agents.ts produces deterministic TOML
- `spawnCodex(request, flags)` in packages/codex-driver/src/spawn/wrapper.ts invokes `codex exec --json` and parses NDJSON

Plan 02-01 is the gluing class. It deliberately stays thin: no extra business logic, no model resolution (that's Plan 02-03), no usage extraction (that's Plan 02-02). The point is to have a concrete `AgentSpawner` instance that methodology can construct + inject.

**`installAgent` semantics**: write to `${CODEX_HOME:-~/.codex}/agents/${role}.toml`. Create parent dirs if missing. Use writeAtomically (already in @swt-labs/artifacts) or local fs writeFile + rename for atomicity. Idempotent — re-installing a role overwrites cleanly. The decision of whether to install vs reuse a user's pre-existing TOML is deferred to Plan 02-03 (where CLI policy lives).

**`spawn` semantics**: pass-through to `spawnCodex(request)`. No flag overrides at this level. Plan 02-03 may inject flags via the constructor if needed.

**`removeAgent` semantics**: unlink the TOML. Idempotent — missing file is a no-op (catch ENOENT). Used by methodology when shutting down a session.

**Tests**: vitest with `mkdtemp` for the codex_home root + execa mock for spawn calls. No real Codex CLI required.
</context>
