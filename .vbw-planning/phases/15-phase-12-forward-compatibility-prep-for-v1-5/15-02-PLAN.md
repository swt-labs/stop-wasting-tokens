### T1 — UI/dashboard tradeoff analysis
Author `.vbw-planning/research/ui-dashboard-tradeoffs.md`:

```markdown
# UI / dashboard design notes (v1.5)

## Context

SWT v1.0 is CLI-only. Multi-week projects accumulate state — current phase, open UAT remediation rounds, telemetry signal, recent failures. Surfacing that state via `swt status` works but doesn't scale to "watch progress live across multiple projects."

A dashboard is the natural v1.5 surface. The decision: **Ink TUI** (terminal-native, in-process), **web** (browser-based, separate process), or **hybrid** (Ink for live + static-site web for archived milestones).

## Option A — Ink TUI

**Pros**
- Zero install — bundled with `@swt-labs/cli`
- No separate process; reads `.swt-planning/` directly
- Terminal-native — fits the SWT user's existing workflow

**Cons**
- Limited rendering (terminal grid, no embedded charts)
- Per-project scope; cross-project monitoring requires running multiple TUIs
- Ink dependency footprint (~5 MB)

**Cost:** S–M (~2 weeks of focused work for a useful first cut)

## Option B — Web

**Pros**
- Rich rendering — charts, mermaid diagrams, multi-project view
- Shareable URLs — onboarding new contributors via link
- Decoupled — could expose a hosted SaaS in v2

**Cons**
- Adds a server process; needs auth if exposed beyond localhost
- Bundle size grows (React + chart lib + Vite output)
- Maintenance burden — separate test pipeline, separate deploy story

**Cost:** L (~4-6 weeks plus operational setup)

## Option C — Hybrid

**Pros**
- Best of both: Ink TUI for live in-progress work, static-site web for shipped milestones (an HTML render of `milestones/<slug>/`)
- Web side is static — no server, just `swt vibe --archive --html` generating Mintlify-style pages
- Existing Mintlify build can render archived milestones natively

**Cons**
- Two surfaces to design + maintain
- Static HTML render still has design overhead

**Cost:** M (~3 weeks if static-only on web side)

## Recommendation

Start with **Option A (Ink TUI) in v1.5** as the live progress surface, scoped to the active milestone. Defer Option B (web dashboard) to v2 once SaaS / multi-project requirements solidify. Add **static HTML rendering of archived milestones** as a v1.5 stretch goal if Ink TUI lands early.

## Decision criteria

If by the end of v1.0 beta:
- Users ask for live cross-project visibility → revisit Option B sooner
- Users mostly run one project at a time → Option A is sufficient
- Users want to share milestone state with stakeholders → static HTML render becomes the priority

The v1.5 milestone scoping should re-evaluate this decision against actual beta feedback.
```

### T2 — canonical v1.5 roadmap
Author `docs/roadmap/v1.5.md` (plain markdown, not MDX). Per-feature template:

```markdown
# v1.5 roadmap

This is the canonical v1.5 roadmap. The docs-site landing page at [`v1-5-roadmap/index.mdx`](../v1-5-roadmap/index) is the user-facing summary; this document is the engineering plan.

## Runtime

### F1 — Real Codex AgentSpawner wiring
**Scope:** Replace `@swt-labs/codex-driver`'s mock AgentSpawner with a live driver wrapping the Codex CLI's subagent-spawn API. Live `swt vibe --execute` runs unblocked.

**Dependencies:** Codex CLI exposes a stable subagent-spawn API.

**Complexity:** L (~3-4 weeks)

**Success criteria:**
- Live `swt vibe --execute` against a real project completes without throws
- Mock driver retained for vitest hermetic runs
- Per-role model resolution honored (resolve-agent-settings.sh equivalent)

### F2 — Claude Code backend driver (`@swt-labs/claude-code-driver`)
**Scope:** Replace the v1.0 stub with a real driver. 12-event hook taxonomy (PreToolUse / PostToolUse / SessionStart / etc.). Agent Teams support. Isolation modes.

**Dependencies:** F1 lessons informing the abstraction layer.

**Complexity:** XL (~6 weeks)

**Success criteria:**
- Same `swt vibe` workflow runs against Claude Code as against Codex
- Hook event taxonomy matches Claude Code's 12 events
- Agent Teams parallelism works the same as Codex teams

### F3 — Ollama backend driver (`@swt-labs/ollama-driver`)
**Scope:** Driver wrapping a local Ollama instance. Self-hosted users.

**Dependencies:** F1.

**Complexity:** M (~3 weeks)

**Success criteria:**
- `swt vibe --execute` against a local Ollama model completes
- Model selection respects `model_overrides` config

## Tooling

### F4 — Ink TUI dashboard
**Scope:** Per-project live dashboard via `swt watch`. Phase progress, agent status, token spend, remediation round count.

**Dependencies:** None (read-only against `.swt-planning/`).

**Complexity:** M (~2 weeks)

**Success criteria:**
- `swt watch` opens a live TUI scoped to the active milestone
- Updates reflect on-disk state changes within 1s
- Closes cleanly on Ctrl+C

See `.vbw-planning/research/ui-dashboard-tradeoffs.md` for the Ink TUI vs web decision.

### F5 — Plugin marketplace integration
**Scope:** Auto-update via `swt update` when the marketplace is the source of truth.

**Dependencies:** Codex Plugin Marketplace API exists.

**Complexity:** S (~1 week)

### F6 — Auto-derived reference docs
**Scope:** Replace hand-authored CLI/config/artifact reference pages with build-time codegen. AUTO-DERIVE-CANDIDATE annotations from v1.0 mark the migration targets.

**Dependencies:** None.

**Complexity:** M (~2 weeks)

**Success criteria:**
- `pnpm docs:gen` produces the three reference pages from source
- Hand-authored "When to override" prose preserved as side-files merged into codegen output
- AUTO-DERIVE-CANDIDATE annotations removed from v1.0 docs once the codegen path is canonical

## Methodology

### F7 — Hook event taxonomy expansion
**Scope:** Add `pre_archive`, `post_phase`, `pre_phase`, `post_uat_fail`, `pre_qa`, `post_qa` hook events alongside the existing `post_archive`.

**Dependencies:** None for the scaffolding; F2 (Claude Code driver) for the load-bearing implementations.

**Complexity:** S (~1 week)

### F8 — Telemetry HTTP sender + hosted endpoint
**Scope:** Replace `NoopSender` default with a real HTTP sender wired to a hosted analytics destination (Plausible / PostHog / self-hosted).

**Dependencies:** Hosted endpoint chosen + procured.

**Complexity:** S–M (~1-2 weeks)

## Compatibility commitment

v1.0 → v1.5 has zero known breaking changes today. Any breaking change introduced between now and v1.5 cut requires:
1. Announcement in this document
2. Migration script in the v1.5 release
3. Major version bump (1.x → 2.0) if the change actually breaks frontmatter or routing

## Beyond v1.5 (v2.0+)

Tracked as "v2 Requirements" in `.vbw-planning/REQUIREMENTS.md`:

- UI/dashboard web (Option B from the design notes)
- Hosted SaaS / cloud orchestration
- Multi-tenant team workflows

These are not promised — they're surfaced so the v1.5 design accounts for them.
```
