<objective>
Wire the three backend drivers (CodexAgentSpawner from Phase 02, ClaudeCodeAgentSpawner from Plan 03-01, OllamaAgentSpawner from Plan 03-03) into a single dispatch surface in the CLI's vibe handler. Add a `backend` config key + matching `--backend` flag override. After this plan ships, `swt vibe --execute` runs the same lifecycle against any of the three backends based on config — the Codex path stays the default, no regression on the v1.5/Phase 02 ship path.
</objective>
<context>
**Decisions resolved 2026-05-07 (user via AskUserQuestion before Phase 03 planning):**
1. **Dispatch mechanism** = single `backend` config key (default `'codex'`).
2. **Per-role overrides:** out of scope for v1.5 (kept simple; deferred to v2 alongside multi-driver orchestration per Phase 02 success criteria).
3. **CLI flag:** `--backend=<value>` works as a one-off override for the same config key.

**Phase 02 / Plan 02-03 baseline:**
- vibe.ts hardcodes `new CodexAgentSpawner()` (line ~92) wrapped in `LazyInstallSpawner` with `getBundledAgentTemplatesDir()` providing the templates path
- `loadSwtConfig` is the public config-loading helper
- `LazyInstallSpawner` accepts any AgentSpawner — backend-agnostic by contract

**Plan 03-05 changes:**
- Add `backend` to ConfigSchema with `z.enum(['codex', 'claude-code', 'ollama']).default('codex')`
- vibe.ts reads `config.backend` (or the CLI flag override) and switches to construct the matching spawner before wrapping in LazyInstallSpawner
- agent-spec-resolver currently resolves a `model` from agents-templates without considering backend — this stays the same for v1.5 (the agent TOMLs declare `model = "gpt-5-codex"` which is Codex-specific). For the claude-code backend the model field maps to a Claude alias like `sonnet` or `opus`; for the ollama backend it maps to a local Ollama model name. **Cross-backend model resolution is a v2 concern.** Plan 03-05 documents this gap as an explicit deviation: when `config.backend !== 'codex'`, the user is responsible for setting `config.model_overrides[role]` to a backend-appropriate model (e.g., `model_overrides.dev = 'claude-sonnet-4-6'` when backend is claude-code).

**`--backend` CLI flag parsing:** add a string flag in `mapFlagsToRouteArgs`. The flag overrides config.backend after the config is loaded, before spawner construction.

**CLI dependency additions:** vibe.ts will import the two new spawner classes, so `packages/cli/package.json` gets `@swt-labs/claude-code-driver: workspace:*` and `@swt-labs/ollama-driver: workspace:*` dependencies (currently only @swt-labs/codex-driver is listed).
</context>
