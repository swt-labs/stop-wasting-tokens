<context>
**Decisions resolved 2026-05-07 (user via AskUserQuestion before Phase 03 planning):**
1. **API surface** = direct `fetch` against `/api/chat` (NDJSON streaming). Zero new runtime deps; uses Node 20+ `globalThis.fetch`.
2. **Out of scope:** `/api/generate` endpoint, hosted Ollama Cloud (per F3 success criterion), and embeddings.

**Ollama `/api/chat` request shape (verified against Ollama 0.x docs):**
```json
{
  "model": "<model-name>",
  "messages": [
    {"role": "system", "content": "<spec.developer_instructions>"},
    {"role": "user", "content": "<request.prompt>"}
  ],
  "stream": true,
  "keep_alive": "5m"
}
```

**Ollama `/api/chat` NDJSON response envelope shape (per line):**
```json
{"model": "...", "created_at": "...", "message": {"role": "assistant", "content": "<text-chunk>"}, "done": false}
```
Final line:
```json
{"model": "...", "created_at": "...", "message": {"role": "assistant", "content": ""}, "done": true, "total_duration": ..., "prompt_eval_count": <input_tokens>, "eval_count": <output_tokens>}
```

**Translation to SwtCore types:**
- text chunks: concatenate `message.content` from each `done:false` line
- usage: derive from `done:true` line's `prompt_eval_count` (→ input_tokens) + `eval_count` (→ output_tokens)
- handoff envelopes: parse from the concatenated text payload (Ollama doesn't have a structured handoff field; the agent must emit the SWT JSON envelope as part of its output text, and the parser re-extracts it via the same Zod schema Phase 02 + Plan 03-01 use)

**Agent profile persistence:** Ollama has no on-disk per-role profile concept (agent profiles are baked into the request). `installAgent(spec)` stores the spec in an in-memory `Map<AgentRole, AgentSpec>` so subsequent `spawn(request)` calls can look up the role's developer_instructions even when the request omits them. `removeAgent` deletes the map entry.

**Pre-existing v1.0 gap:** `packages/ollama-driver/` has NO `tsconfig.json`. Plan 03-03 lands a tsconfig matching codex-driver's pattern.
</context>
