# Phase 9 / Plan 06: Verify mode inline UAT checkpoint loop + Milestone UAT recovery

## Why sixth

PLAN 05 shipped the artifact-write side of verification. PLAN 06 ships the **interaction** side: the inline UAT checkpoint loop that asks the user for PASS/FAIL on each test row, captures issue records, and respects the configured remediation round cap. It also adds the milestone UAT recovery handler so archived milestones with unresolved UAT issues route correctly. After this lands, `swt vibe` can drive a phase from execute → qa → verify → re-verify → archive (modulo PLAN 07's audit gate, which is the only piece left before milestone archive works end-to-end).

## Tasks

### T1 — core/Prompter abstraction
`Prompter` interface with `askChoice<T extends string>({prompt, options, default?})`,
`askText({prompt, default?})`, `askConfirm({prompt, default?})`. All return Promise.

### T2 — ScriptedPrompter (test fixture)
`ScriptedPrompter` constructor takes a `(prompt) => answer` script or a
`Map<string, unknown>`-style queue. Used by tests for deterministic runs.

### T3 — round-cap resolver
`resolveUatRemediationRoundLimit({config, currentRound})` returns
`{nextRound, maxRounds, capReached}`. `config.max_uat_remediation_rounds === false`
is unlimited; positive integer caps; missing defaults to false.

### T4 — verifyHandler inline checkpoint loop
Add `prompter?: Prompter` to `VerifyHandlerOptions`. When provided and
`autonomy !== 'pure-vibe'`, run the checkpoint loop: for each synthesized row,
ask PASS/FAIL/SKIP/DEFER; on FAIL, ask issue summary + severity; build the
UatDoc.tests[] and issue_records[] arrays accordingly. Aggregate `passed`,
`skipped`, `issues`, and `status`. When no prompter or autonomy is pure-vibe,
keep today's deferred-row default behavior (so existing tests still pass).

### T5 — reVerifyHandler honors round cap
Pre-check the configured cap before archiving the prior UAT. When
`capReached=true`, write a banner to stdout (`Reached maximum UAT
remediation rounds…`), do not move the prior UAT, do not bump remediation
state, return `HandlerResult{exit:0}`.

### T6 — milestoneUatRecoveryHandler
`milestoneUatRecoveryHandler({prompter, today?})` scans
`.swt-planning/milestones/<latest>/phases/<NN>-<slug>/<NN>-UAT.md`,
collects unresolved issues, presents a `RecoveryDecision` choice
(create remediation phases | start fresh | skip). On `start-fresh`, writes
`.remediated` markers and clears the routing trigger. On `create-remediation`,
returns a `HandlerResult{exit:0, ranTo:'completion'}` with a message that the
caller (CLI) uses to dispatch the Add Phase flow (PLAN 06 doesn't yet wire
that flow end-to-end — that's a smaller follow-up to compose with the existing
add-phase artifacts when the route lands in CLI).

### T7 — Wire into vibe registry + CLI
Update `packages/cli/src/commands/vibe.ts` to register the new handler and to
construct a `ReadlinePrompter` (or NoopPrompter when stdin is not a TTY) for
`verifyHandler` and `milestoneUatRecoveryHandler`.

### T8 — Vitest
- `round-cap.test.ts`: unlimited / under cap / at cap / past cap.
- `verify.test.ts` (extended): scripted prompter all-pass; one-fail with issue capture; defer path.
- `re-verify.test.ts` (extended): cap-reached path surfaces banner and does not bump round.
- `milestone-uat-recovery.test.ts`: scan + decision matrix (create / start-fresh / skip).

### T9 — Commit + summary
`feat(methodology): verify inline checkpoints + milestone uat recovery + round cap (Phase 9 / PLAN 06)`.
