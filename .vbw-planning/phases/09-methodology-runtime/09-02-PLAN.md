# Phase 9 / Plan 02: Orchestration loop

## Why second

PLAN 01 ships `detectPhase` (the data). PLAN 02 ships the routing layer (the dispatcher). With both in place, every subsequent plan only needs to fill in one mode handler at a time — the scaffolding is complete.

## Layout

```
packages/methodology/src/vibe/
├── index.ts
├── route.ts                       # VibeRoute discriminated union + routeFromState()
├── handlers/
│   ├── index.ts                   # ModeHandler interface, registry
│   ├── stubs.ts                   # NotImplementedError stub factory
│   └── (real handlers added in PLAN 03+)
└── errors.ts                      # NotImplementedError, RoutingError
packages/methodology/test/vibe/
├── route.test.ts                  # priority table cell-by-cell
└── dispatch.test.ts               # registry + stub propagation
packages/cli/src/commands/vibe.ts  # real implementation, replaces stub
```

## Tasks

### T1 — errors.ts
Two error classes: `NotImplementedError` (`code='mode_not_implemented'`, carries `mode: string` + `roadmap_pointer: string`) and `RoutingError` (`code='routing_error'`, carries the offending state for diagnostics).

### T2 — route.ts
- `VibeRoute` discriminated union with `kind` field per mode (`init-redirect`, `bootstrap`, `scope`, `discuss`, `plan-and-execute`, `execute`, `verify`, `qa-remediation`, `uat-remediation`, `re-verify`, `milestone-uat-recovery`, `archive`, `all-done`, `add-phase`, `insert-phase`, `remove-phase`).
- `RouteArgs` for CLI flag overrides (`--plan`, `--execute`, etc.).
- `routeFromState(state, args)` implements the VBW priority order:
  1. `planning_dir_exists=false` → init-redirect
  2. `project_exists=false` → bootstrap
  3. `next_phase_state=needs_uat_remediation` → uat-remediation
  3.5. `next_phase_state=needs_qa_remediation` → qa-remediation
  4. `next_phase_state=needs_reverification` → re-verify
  5. `milestone_uat_issues=true` → milestone-uat-recovery
  6. `phase_count=0` → scope
  7. `next_phase_state=needs_verification` → verify (with QA-pending sub-flag)
  8. `next_phase_state=needs_discussion` → discuss
  9. `next_phase_state=needs_plan_and_execute` → plan-and-execute
  10. `next_phase_state=needs_execute` → execute
  11. `next_phase_state=all_done` → archive (with QA-attention fallback to verify)
- The all_done QA-attention fallback (pending) and earlier-work QA-attention fallback (failed) implemented as documented in vibe.md.

### T3 — handlers/index.ts
- `ModeHandler<TArgs>` interface: `{ kind: VibeRoute['kind']; canHandle: (route) => boolean; run: (route, io) => Promise<HandlerResult> }`.
- `HandlerResult = { exit: ExitCode; route: VibeRoute; ranTo: 'completion' | 'stub' }`.
- `ModeRegistry` with `register(handler)` and `dispatch(route, io)`.

### T4 — handlers/stubs.ts
`stubHandler(kind, roadmap_pointer)` returns a `ModeHandler` whose `run` throws `NotImplementedError`. Used to populate the registry for every mode that doesn't yet have a real handler. The registry exposes a `populateStubs(modes, pointer)` helper.

### T5 — vibe.ts CLI command
Replace the stub. New flow:
1. Run `detectPhase()`.
2. Resolve route via `routeFromState(state, parsedArgs)`.
3. Dispatch through the registry.
4. On `NotImplementedError`, print: "Route resolved: {kind}\nNot yet implemented (Phase 9 / Plan {NN}: {roadmap_pointer}).\nRun \`swt detect-phase\` for state details." Return exit 2.
5. On `RoutingError`, print error + state JSON. Return exit 1.
6. On success, return whatever `HandlerResult.exit` says.

### T6 — Vitest
- `route.test.ts`: each priority cell — synthetic `PhaseDetectResult` → expected `VibeRoute.kind`. Covers all 11 priorities + the QA-attention fallbacks.
- `dispatch.test.ts`: registry registers + dispatches; stub throws `NotImplementedError`; CLI surface returns exit 2 on stub.
