# Phase 9 / Plan 07: Archive + 7-point audit gate

## Why seventh

PLAN 06 closed the verification loop. PLAN 07 closes the milestone — once a phase set is fully built and verified, the runtime needs to deterministically gate the archive (UAT + state consistency + 7 audit points), derive a slug, and move planning artifacts into `milestones/{slug}/`. After this lands, `swt vibe` covers the full lifecycle bootstrap → scope → plan → execute → qa → verify → re-verify → archive (with the Discussion engine still deferred to PLAN 03b for the interactive bootstrap/scope flows).

## Tasks

### T1 — artifacts/milestones/derive-slug.ts
`deriveMilestoneSlug({planningDir})` reads `ROADMAP.md`, extracts phase entries (e.g. `## Phase 1: Setup`), kebab-cases each name, and joins them with the milestone position to form `01-setup-api-layer`-style slugs. Falls back to `milestone-{date}` when no phase headings are found.

### T2 — methodology/audit/audit.ts
`runArchiveAudit({planningDir})` returns an `AuditResult` with seven entries:
1. roadmap_completeness — every roadmap phase has a non-TBD goal.
2. phase_planning — every phase dir has ≥1 PLAN.md.
3. plan_execution — every PLAN.md has a matching SUMMARY.md.
4. execution_status — every SUMMARY.md frontmatter has `status: complete`.
5. verification — every phase dir has VERIFICATION.md with `result: pass` (warn when missing, fail when `result: fail`).
6. uat_status — every phase dir has UAT.md and none have `status: issues_found`.
7. requirements_coverage — every roadmap REQ-ID exists in REQUIREMENTS.md.

Fail = block. Warn = proceed with warnings. The handler honors `--skip-audit` flag for non-UAT checks.

### T3 — methodology/audit/uat-guard.ts
`runArchiveUatGuard({planningDir})` returns `'pass' | 'fail'`. Scans active `.swt-planning/phases/` and the latest `.swt-planning/milestones/{slug}/phases/` (without `.remediated` markers). Any UAT.md with `status: issues_found` or `issues > 0` fails the guard. Non-bypassable.

### T4 — methodology/audit/state-consistency.ts
`runStateConsistencyCheck({planningDir})` cross-checks:
- `STATE.md` `phase_count` (frontmatter or "Phases: N" line) vs `phases/` dir count.
- Every PLAN.md has matching SUMMARY.md.
- Every SUMMARY.md frontmatter has `status: complete`.
Returns `{ok: boolean, failures: string[]}`. Non-bypassable.

### T5 — handlers/archive.ts
`archiveHandler({skipAudit?, force?, today?})` runs the gates in order: UAT guard → state-consistency → audit (skip when `skipAudit && !uatFail`). Derives slug, moves phase artifacts via `fs.rename`, writes `.swt-planning/milestones/{slug}/SHIPPED.md` with frontmatter (slug, shipped_at, phase_count, requirement_count). Persists project-level STATE.md sections (Todos, Decisions, Blockers, Codebase Profile) back to root via a small extractor.

### T6 — handlers/all-done.ts
`allDoneHandler` returns a friendly message when routing reports `kind='all-done'`. No-op handler so the registry has full kind coverage.

### T7 — Wire into vibe registry
Update `packages/cli/src/commands/vibe.ts` to register `archiveHandler` and `allDoneHandler`.
