<context>
Phase 15 of v1.0 added `agents-templates/*.toml` (6 files) with `model = "gpt-5.5-pro"` — that model does not exist. A user running `codex exec --profile scout` against a freshly-initialised SWT project gets a model-not-found error. The fix is to either pin a real Codex-supported model (e.g. the canonical Codex-default at v1.0 ship — likely `gpt-5-codex` or similar), or set `model = "default"` / leave the key out so Codex picks from the user's `~/.codex/config.toml` profile.

The Codex Plugin Marketplace manifest at `packages/cli/codex-plugin.json` line 2 has `"$schema": "https://docs.codex.example/plugin-manifest.schema.json"` — `docs.codex.example` is RFC-2606 reserved (never resolves). The marketplace will reject the manifest. Either swap to the real schema URL (when it's known) or remove the `$schema` field entirely if it's optional in the marketplace spec.

`allowed_mcp_servers = ["filesystem", "web-fetch", "web-search"]` in the agent TOMLs uses generic descriptors, not actual MCP server identifiers. Two viable fixes: (a) replace with real, well-known MCP server names if they exist for these capabilities, or (b) leave the descriptors and add a TOML-level comment marking them as illustrative placeholders that users must override per their own MCP setup.
</context>
