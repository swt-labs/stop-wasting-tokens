<objective>
Make SWT's 6 agent profile TOMLs loadable by a real Codex CLI by:
1. Replacing the fictional `gpt-5-codex` model identifier with documented Codex models (`gpt-5.3-codex` for coding-tuned roles; `gpt-5.5` for general roles).
2. Replacing SWT's Effort-tier values (`balanced`, `thorough`) used in the `model_reasoning_effort` TOML field with Codex's documented enum (`minimal | low | medium | high | xhigh`). This requires decoupling SWT's internal Effort concept from Codex's reasoning-effort concept by introducing a new `CodexReasoningEffort` type in @swt-labs/core.
3. Adding the Codex-required `name` and `description` fields to every template, per `developers.openai.com/codex/subagents`.

This closes the F-01, F-02, and F-04 critical findings from the v1.5.1 milestone scope. Without this fix, no SWT-spawned agent profile would be accepted by a real Codex CLI today.
</objective>
<context>
**Source research:** `.vbw-planning/research/20260507-081032-verify-codex-sdk-best-practices.md` — F-01 (CRITICAL), F-02 (CRITICAL), F-04 (HIGH).

**Current state (per the research):**
- Each of the 6 agent TOMLs declares `model = "gpt-5-codex"` (fictional — Codex's actual coding model is `gpt-5.3-codex`).
- Each declares `model_reasoning_effort = "balanced"` (a SWT Effort-tier value, not a Codex reasoning_effort enum value).
- None declare `name` or `description` (both required per Codex subagent schema).

**Why decouple Effort and CodexReasoningEffort:**
SWT's existing `Effort` type (`thorough/balanced/fast/turbo`) lives at `packages/core/src/types/effort.ts` and is the planning-depth + turn-budget tier — it's used by the config layer (`config.json:effort`) and the agent_max_turns scalar machinery. It is a **SWT-internal** concept.

Codex's `model_reasoning_effort` is the model's thinking budget — a Codex CLI runtime config (`minimal | low | medium | high | xhigh`) per `developers.openai.com/codex/config-reference`. It is a **Codex-runtime** concept.

These two concepts share a TOML field name (`model_reasoning_effort`) by accident in v1.5; the fix is to introduce a separate `CodexReasoningEffort` type and use it for the AgentSpec.reasoning_effort field + the TOML validator. SWT's internal Effort tier stays untouched.

**Per-role assignment (deliberate, per research recommendation):**
- `scout` — `model = "gpt-5.5"`, `reasoning_effort = "low"` (read-only investigation; light reasoning suffices)
- `architect` — `model = "gpt-5.5"`, `reasoning_effort = "high"` (design-heavy work benefits from high reasoning + frontier model)
- `lead` — `model = "gpt-5.3-codex"`, `reasoning_effort = "medium"` (planning over coding-tuned base)
- `dev` — `model = "gpt-5.3-codex"`, `reasoning_effort = "medium"` (coding workhorse)
- `qa` — `model = "gpt-5.3-codex"`, `reasoning_effort = "medium"` (verification over coding-tuned base)
- `debugger` — `model = "gpt-5.3-codex"`, `reasoning_effort = "high"` (deep investigation needs both coding-tuned + high reasoning)

**Backwards compat / breakage scope:**
- Plan 01 changes the template TOML schema. Any user with overrides referencing the old `model_reasoning_effort = "balanced"` etc. will get a clear ConfigError pointing at the new enum. This is acceptable because v1.5 hasn't shipped to real users and the templates ARE the source of truth.
- The `role` field stays in the templates (deferred to Tier 4 / F-07). `name` is added as a separate Codex-required field; SWT continues to use `role` internally.
- Existing `allowed_mcp_servers` and `max_turns` fields remain (deferred to Tier 4 / F-05, F-06).

**Out of scope:**
- Plan 01 does NOT touch the codex-driver's hooks-writer (Phase 03 territory).
- Plan 01 does NOT move the plugin manifest (Phase 02 territory).
- Plan 01 does NOT update the F-08 TOML header comment about `~/.codex/mcp.json` path (Phase 03 covers it).
</context>
