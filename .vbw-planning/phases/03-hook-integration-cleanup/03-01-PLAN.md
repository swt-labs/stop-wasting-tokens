<objective>
Wire the codex-driver's hooks emit path so SWT's HookHost taxonomy correctly translates to Codex's documented `hooks.json` schema. Specifically: (1) translate snake_case event names → PascalCase per the Codex docs, (2) wrap SWT's flat event arrays in Codex's nested `hooks.{EventName}: [{matcher, hooks: [{type, command, timeout}]}]` shape, (3) emit a `[features] codex_hooks = true` TOML block for the user's `~/.codex/config.toml` to enable hooks. Cleans up doc/comment drift in the agent TOML headers (F-08).

This closes F-08 (MEDIUM), F-09 (MEDIUM), F-10 (MEDIUM), F-11 (MEDIUM) from the v1.5.1 milestone scope.
</objective>
<context>
**Source research:** `.vbw-planning/research/20260507-081032-verify-codex-sdk-best-practices.md` — F-08, F-09, F-10, F-11.

**Codex `hooks.json` documented schema** (per `developers.openai.com/codex/hooks`):
```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "regex_pattern_or_tool_name",
        "hooks": [
          {
            "type": "command",
            "command": "script path or command",
            "statusMessage": "optional UI message",
            "timeout": 600
          }
        ]
      }
    ]
  }
}
```

**Current SWT `hooks/writer.ts` schema** (flat, snake_case):
```json
{
  "session_start": [{"command": "...", "match": "...", "cwd": "..."}],
  "user_prompt_submit": [...],
  ...
}
```

**Translation strategy:**
- Keep SWT's `HookFile` interface unchanged (backward compat — internal callers still use snake_case keys + flat shape).
- Add a new `emitCodexHooksJson(file: HookFile): string` function that translates SWT → Codex schema at emit time:
  - snake_case key → PascalCase (per the documented translation map).
  - Each entry's `match` field → `matcher` (renamed per Codex docs).
  - Each entry's `command` → wrapped in Codex's nested `hooks` array with `type: "command"`.
  - Default `timeout: 600` per Codex docs.
- Filtering of v1.5 SDLC events (pre_archive, post_phase, etc.) is implicit: SWT's HookFile interface only has the 6 v1.0 keys, so emit-time translation cannot accidentally include v1.5 keys.

**Feature flag emission:**
- Codex docs require `[features] codex_hooks = true` in the user's `~/.codex/config.toml` to enable the experimental hooks feature.
- Add `emitCodexHooksFeatureFlag(): string` that returns `"[features]\ncodex_hooks = true\n"` for callers (e.g., the codex-driver installer) to merge into the user's config.toml.
- Wiring this into the install path (codex-agent-spawner.ts or a new install hook) is OUT OF SCOPE for Plan 03-01 — exporting the function is the must_have. The install wiring is tracked as a v1.5.1 follow-up.

**TOML header comment fix (F-08):**
The current header comment in all 6 agent templates says: *"`allowed_mcp_servers` below lists illustrative identifiers — replace with the real MCP server names registered in your `~/.codex/mcp.json` (or omit servers you don't have configured)."* Codex's actual MCP config path is `~/.codex/config.toml` with `[mcp_servers.<server-name>]` blocks, NOT a separate `~/.codex/mcp.json` file. Update the header comment in all 6 TOMLs.

**Out of scope:**
- F-12 expansion of HookSubBlockSchema in @swt-labs/core (tier 4 / v1.6+ deferred): SWT's internal `ConfigSchema.hooks` block stays as `{script_path: string}`. Plan 03-01 only changes the codex-driver's *emit* path, not the SWT-internal config schema.
- Wiring `emitCodexHooksFeatureFlag` into the codex-driver's install path: out of scope — the function is exported, but the install-time merge into `~/.codex/config.toml` is a v1.5.1 follow-up.
- Methodology-side dispatch firing each new SDLC event from the right command surface (separate v1.5 follow-up tracked since Phase 05).
