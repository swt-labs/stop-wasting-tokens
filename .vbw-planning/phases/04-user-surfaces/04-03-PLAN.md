<objective>
Replace the v1.0 `NoopSender` default with a real `HttpSender` that POSTs telemetry events to a configurable endpoint behind the existing `enabled: true` opt-in. Preserve the v1.0 privacy contract exactly: no PII, anonymous UUIDv4, sanitize-on-send (sanitize is already done by TelemetryClient before events reach the sender), retry-once on transient failures, never block or crash the user's session.
</objective>
<context>
**Decision resolved 2026-05-07 (user via AskUserQuestion before Phase 04 planning):**
- HTTP client = `globalThis.fetch` (Node 20+ built-in). No new runtime deps.
- Retry policy = retry-once on network error or 5xx; no retry on 4xx; drop silently after retry exhaust.
- Timeout = 5 seconds via AbortSignal.

**HttpSender shape:**
```typescript
export interface HttpSenderOptions {
  readonly endpoint: string;
  readonly fetchImpl?: typeof globalThis.fetch;
  readonly timeoutMs?: number;
  readonly onWarning?: (msg: string) => void;
}

export class HttpSender implements Sender {
  constructor(opts: HttpSenderOptions);
  async send(events: readonly TelemetryEvent[]): Promise<void>;
}
```

**Send flow:**
1. POST `${endpoint}` with body `JSON.stringify({events})`, `Content-Type: application/json`, AbortSignal timeout 5s.
2. 2xx → success, return.
3. 5xx → wait 1s (jittered ±200ms), retry once. If second attempt fails, log warning and drop silently.
4. 4xx → no retry; log warning and drop silently.
5. Network error / abort / timeout → wait 1s (jittered), retry once. If second attempt fails, drop silently.

**ConfigSchema change** in `packages/core/src/config/Config.ts`:
```typescript
telemetry: z.object({
  enabled: z.boolean().default(false),
  anonymous_id: z.string().uuid().optional(),
  opted_in_at: z.string().optional(),
  endpoint: z.string().url().optional(),
  cache_ttl_hours: z.number().int().positive().default(24),
}).default({ enabled: false }),
```

(`cache_ttl_hours` is documented in the F8 success criterion but isn't actually consumed by HttpSender today — it's a forward-compatible config field for future client-side dedupe-window use. F8's privacy contract doesn't require it for v1.5.)

**Integration with TelemetryClient:** TelemetryClient stays unchanged — its Sender field is generic over the Sender interface. CLI startup (or test wiring) chooses which sender to construct:
```typescript
const sender = config.telemetry.enabled && config.telemetry.endpoint
  ? new HttpSender({ endpoint: config.telemetry.endpoint })
  : new NoopSender();
```

CLI integration is OUT OF SCOPE for this plan — it's a cross-feature concern that lands when telemetry is wired into actual command surfaces. Plan 04-03 ships the HttpSender class + its tests. The factory-style construction at the CLI boundary is a v1.5 follow-up tracked alongside the broader telemetry rollout.

**No-PII contract:**
- HttpSender does not read `process.env.USER` / `os.userInfo()` / etc.
- No User-Agent override (the default Node fetch User-Agent doesn't carry identifiers).
- No auth headers (telemetry is anonymous-by-design).
- The events themselves are already sanitized by TelemetryClient via `sanitize.ts` before they reach the sender.
