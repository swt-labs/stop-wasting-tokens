## Tasks

### T1 вЂ” Six agent TOMLs
Each TOML is an `AgentSpec` rendered as TOML вЂ” role, model, reasoning effort, sandbox mode, allowed MCP servers, developer instructions. Modelled per the source plan's role responsibilities (Scout = research, Architect = design, Lead = planning, Dev = implementation, QA = verification, Debugger = root-cause investigation with `xhigh` reasoning).

### T2 вЂ” Six skills
Each SKILL.md has YAML frontmatter `name` + `description` (trigger-rich, в‰Ґ 10 chars) and a body that explains when the skill auto-matches and what it produces. Skills are the primary user-facing methodology delivery вЂ” ergonomics matter.

### T3 вЂ” Profile resolvers
Three small modules exposing the preset в†’ resolved-config mapping for effort, autonomy, and verification. Pure functions; no I/O.

### T4 вЂ” Cache-aware prompt builder
`buildPrompt(config)` returns `{ prefix, suffix, prefixHash }`. The prefix is the cache-friendly static layer (role, instructions, project conventions). The suffix is the dynamic per-call layer. `hashPrefix(prefix)` returns a SHA-256 hex string for cache-stability assertions.

### T5 вЂ” MEMORY.md model
`FileMemoryStore` implements `MemoryStore` from core:
- root file at `<dir>/MEMORY.md` (the always-on lightweight index)
- topic files at `<dir>/memory/<topic>.md`
- entries serialised as YAML frontmatter + body
- index regenerated on `compact()`

### T6 вЂ” Vitest suite
- profiles.test.ts вЂ” preset values, override merging.
- prompt-builder.test.ts вЂ” same config в†’ identical prefix and identical hash.
- memory.test.ts вЂ” put/get/query/compact round-trip on a temp dir.

### T7 вЂ” Commit + summary
`feat(methodology): agents, skills, prompt builder, profiles, memory store`.
